# Developer Communication Examples

This directory contains realistic filled examples for the practice artifacts.

Use them after you finish your own first draft.

The goal is not to memorize wording.

The goal is to recognize what strong structure, explicit reasoning, and honest trade-off communication look like when the artifact is complete.

## Included Examples

- `issue-writing-example.md`
- `pr-description-example.md`
- `adr-example.md`
- `review-reply-example.md`
- `speaking-about-systems-example.md`

Each example now shows a weak version first and then a stronger version.

This makes it easier to see not only what good looks like, but how quality improves.

## How To Use

1. complete the matching exercise first
2. score your own version
3. compare it with the example
4. note where the example is more explicit, more actionable, or more honest about uncertainty
5. rewrite your version once more without copying the exact phrasing

Pay special attention to what the weak version gets wrong:

- missing scope
- fake certainty
- vague risk language
- missing trade-offs
- structure that forces unnecessary follow-up questions

## Rule

Reference examples are calibration tools.

They are not scripts to paste into real work without adapting the facts to the actual situation.

# AI Learning

This section is a full operating system for accelerated learning and AI-assisted development.

Its goal is not to replace hard thinking.

Its goal is to help you move from beginner to architecture-level reasoning faster, with better workflows, stronger validation, and more deliberate tool use.

## What This Section Is For

Use `ai-learning/` to learn how to:

- study technical topics faster without becoming shallow
- use AI tools correctly during coding, debugging, design, and research
- choose the right tool for the right task instead of following hype
- write better issues, PR notes, architecture summaries, and technical explanations
- improve English and other foreign-language skills in ways that help developer work directly
- build architecture thinking rather than only prompt-writing habits

## Core Principle

This section is built around one rule:

AI should compress friction, not outsource understanding.

If AI makes you faster but weaker, the workflow is bad.

If AI makes you faster and more precise, the workflow is good.

## Who This Section Is For

This section is useful if you are:

- learning from zero and need structure
- already coding but want to work faster without losing quality
- moving from implementation into architecture thinking
- trying to use AI tools professionally rather than casually
- learning English for technical reading, writing, speaking, and collaboration

## Operating Model: From Zero To Architect

The progression in this section is intentional.

### Level 1 - Beginner

Focus:

- ask better questions
- understand concepts
- avoid copy-paste learning
- turn AI answers into active recall and practice

### Level 2 - Builder

Focus:

- use AI for implementation support
- debug more systematically
- generate tests, examples, and first drafts
- review outputs critically

### Level 3 - Systems Thinker

Focus:

- compare design alternatives
- reason about boundaries, contracts, and trade-offs
- use AI for research and structured decision-making
- write clearer technical artifacts

### Level 4 - Architecture Mindset

Focus:

- use AI as a design amplifier, not a substitute
- evaluate cost, risk, complexity, and team impact
- separate short-term speed from long-term maintainability
- convert tool output into sound engineering judgment

## Real Examples

This section should help with concrete outcomes, not generic inspiration.

Examples:

- ask AI to explain why a Spark partitioning strategy is risky
- ask AI to review a dbt model for missing tests and grain confusion
- ask AI to compare Delta Lake versus Iceberg for a multi-engine use case
- use AI to draft a PR summary and then tighten it into a precise engineering note
- use English-learning tools to improve issue writing, architecture notes, and spoken explanations of systems

## Example: Learning Loop With Code

Bad pattern:

```python
# Generated code accepted without understanding.
def transform(records):
		return [record["value"] * 2 for record in records]
```

Better pattern:

```python
def transform(records):
		transformed = []
		for record in records:
				if "value" not in record:
						raise ValueError("Missing required field: value")
				transformed.append(record["value"] * 2)
		return transformed
```

Then use AI for the next step:

- ask what edge cases are missing
- ask what tests should exist
- ask whether this behaviour should fail fast or skip invalid rows

This is the difference between raw generation and architecture-minded iteration.

## Best Practices

- Think once yourself before asking AI.
- Ask narrower questions.
- Ask for trade-offs and failure modes, not only solutions.
- Separate exploration from execution.
- Validate claims against code, docs, or actual outputs.
- Keep one source of truth for decisions and notes.
- Use AI to improve reasoning, not to remove it.
- Build workflows, not random prompt habits.

## What Not To Do

- Do not ask AI to replace first-principles thinking.
- Do not treat long answers as correctness.
- Do not use one tool for every task.
- Do not let AI-generated code bypass review and testing.
- Do not turn English learning into passive translation only.
- Do not optimize for novelty of tools over repeatability of workflow.

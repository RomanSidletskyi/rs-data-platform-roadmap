# Developer Communication

This directory focuses on technical communication that directly affects engineering quality.

The goal is not to write more.

The goal is to communicate more clearly, more precisely, and with better engineering judgment.

## Why This Matters

Strong engineering is not only implementation.

It also depends on how well you can:

- explain a bug
- describe a change
- capture assumptions and trade-offs
- ask for feedback
- respond to review
- present a design clearly in speech or writing

Poor communication creates real technical cost:

- unclear requirements
- weak reviews
- hidden assumptions
- bad handoffs
- wasted debugging time

## What This Directory Covers

- writing better issues
- writing clearer PR descriptions
- writing architecture notes that explain trade-offs
- communicating well in design reviews
- speaking about systems clearly in English or mixed-language environments
- comparison guides for issue, PR, ADR, review, and speaking patterns

## Practice Layer

The comparison guides help you recognize stronger and weaker patterns.

The `practice/` subdirectory is where you actually train those patterns.

Use it to rehearse:

- issue writing
- PR descriptions
- ADR summaries
- review replies
- spoken system explanations

## How To Use This Directory

Use the files in two passes:

1. read the main guide for the artifact type you want to improve
2. read the matching comparison guide to see weak, acceptable, and strong patterns side by side

Use `comparisons-by-context.md` when you are not sure which artifact is appropriate for the situation in the first place.

This is especially useful when you already "kind of know" how to communicate, but the quality is inconsistent.

Then move into `practice/` and produce your own versions.

## Working Rule

Strong technical communication is usually structured, explicit, and honest about uncertainty.

AI can help improve clarity and tone.

It must not be allowed to hide missing reasoning, missing evidence, or unresolved risk.

Write the substance yourself first.

Then use AI to tighten the message.
